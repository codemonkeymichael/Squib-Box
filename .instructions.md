# Squib-Box DMX Receiver Project

## Overview
DMX receiver system using Raspberry Pi Pico + MAX485 RS485 transceiver. Receives DMX512 commands from any standard DMX controller (Enttec, Chauvet, ETC, etc.) via XLR connector.

## Hardware Setup

**Data Flow:** Standard DMX controller (XLR output) → MAX485 transceiver (RS485→UART converter) → Pico UART1

**Pico UART1 Configuration:**
- Baud rate: 250,000 bps (standard DMX512)
- Data: 8 bits, no parity, 2 stop bits
- RX pin: GPIO 5 (connect to MAX485 RO - Receiver Output)

**MAX485 Module Wiring:**
- **RO (Receiver Output)** → Pico GPIO 5 (UART1 RX)
- **DE (Driver Enable)** → GND (receive-only mode, disable transmitter)
- **RE (Receiver Enable)** → GND (enable receiver)
- **A & B (Differential pair)** ← From Enttec XLR connector (3-pin or 5-pin)
- **Power: 3.3V** from Pico (can also use external 5V if module supports it)
- **Ground: Common** with Pico

**CRITICAL: A/B Polarity**
- Enttec pin A → MAX485 pin A
- Enttec pin B → MAX485 pin B
- **Reversed polarity causes all received bytes to read as 0x00** (all bits flipped at RS485 level)
- Verify with test: If all channels show 0, swap the A/B wires at the MAX485 connector

## GPIO Assignments (Triggers)
- **GPIO 2**: Channel 1 (falling edge / turn OFF)
- **GPIO 3-4, 6-18**: Channels 2-16 (rising edge / turn ON)
- **GPIO 5**: Reserved (UART RX - DO NOT USE)

## Key Technical Findings

### Frame Structure & DMX Break Encoding
- **Standard DMX512 protocol**: Break signal + Start Code (0x00) + 512 channel bytes
- **At the Pico UART**: The DMX break (long LOW signal ~88µs) gets encoded by the UART as ~13 zero bytes
- **Resulting frame in buffer**: [13 x 0x00 break] + [0x00 start code] + [ch1] [ch2]...[ch16]
- **Channel extraction offset**: +13 bytes from the start code (skips the break bytes and points to ch1)

**Why this matters:**
- The MAX485 faithfully converts the RS485 differential signal to UART
- The UART interprets the DMX break (LOW voltage) as 0x00 bytes
- This is normal and expected—not a bug
- The break byte count is consistent: typically 13 bytes on RP2040 at 250kbaud

### Timing-Based Detection (ABANDONED)
- **Original approach**: Measure time between byte arrivals, detect inter-frame gaps
- **Problem**: Pico polling loop runs at ~20µs, but DMX inter-frame gaps are ~2000µs → timing measurements unreliable
- **Result**: Buffer overflow ("Discarded old data" errors)
- **Solution**: Replaced with pattern-matching on protocol markers (robust and polling-speed independent)

### Debouncing Implementation
- **3-frame circular buffer** for channel history
- **Threshold validation**: All 3 consecutive frames must agree (all >100 or all ≤100) before state change triggers
- **Rising-edge detection** for channels 2-16: Only fires when transitioning from OFF to ON
- **Level control** for channel 1: Continuous HIGH when >100, continuous LOW when ≤100
- **Debounce delay**: ~35-70ms depending on frame rate (imperceptible in theater operation)

### Channel Behavior
- **Channel 1 (GPIO 2)**: Level control
  - GPIO HIGH when channel value > 100
  - GPIO LOW when channel value ≤ 100
  - Continuous output (follows fader position)
  
- **Channels 2-16 (GPIO 3-4, 6-18)**: Rising-edge pulse triggers
  - GPIO fires 250ms pulse when transitioning from ≤100 → >100
  - Repeats only on subsequent rising edges (low to high transitions)
  - Does NOT fire on falling edges or sustained HIGH

## Current Script Behavior (dmx_receiver.py)
- **Pattern-based frame detection** (FINAL WORKING SOLUTION): 
  - Finds first non-zero byte in buffer (guaranteed to be real channel data, not break bytes)
  - Backtracks one position to locate the 0x00 start code marker
  - This elegantly skips the DMX break (~13 encoded zero bytes at UART level)
  - Extracts 16 channel bytes immediately following the start code
- **Standard DMX512 offset**: Reads channels at +1 from start code (after accounting for frame alignment)
- **3-frame debounce validation**: Only triggers on state change when all 3 frames agree
- **Dual-core operation**: 
  - Core 0: DMX receive + debounce + queue triggers
  - Core 1: Execute 250ms pulses asynchronously
- **Channel mapping**:
  - Ch1 (GPIO 2): Level control (continuous HIGH/LOW)
  - Ch2-16 (GPIO 3-4, 6-18): Rising-edge pulse triggers (250ms each)

## Relay Wiring
Each GPIO output needs a relay circuit (typical setup):
```
GPIO → 470Ω resistor → Transistor base (or relay coil directly if low-power)
Relay coil → +12V supply (or appropriate voltage)
Relay coil → Flyback diode (1N4007) to GND
Relay contacts → Load (lights, motors, etc.)
```

## Development History & Learnings (Session 2026-07-11/12)

### Problem 1: Buffer Overflow with Timing-Based Detection
- **Symptom**: "BUFFER OVERFLOW: Discarded old data" errors; frames never detected as complete
- **Root cause**: Original code used FRAME_IDLE_US threshold to detect inter-frame gaps by measuring time between polling attempts. Pico polling loop runs at ~20µs, but inter-frame gaps are ~2000µs. Timing measurements unreliable at high polling rates.
- **Solution**: Replaced with pattern-based frame detection (search for `0x00 0x00` marker)
- **Lesson learned**: Timing-based detection fundamentally incompatible with fast polling loops

### Problem 2: All Channels Reading as 0x00
- **Symptom**: Pattern detection working, frames extracting, but every byte = 0x00. QLC+ lights responding correctly (proving transmission worked)
- **Root cause**: MAX485 A/B differential pair wires swapped at connector or pin order reversed
- **Solution**: User physically swapped A/B signal wires on MAX485 transceiver → data immediately restored
- **Lesson learned**: RS485 polarity inversion corrupts data to all zeros (all bits flipped); check wiring first

### Problem 3: Frame Extraction Misalignment
- **Symptom**: Real channel data (5B 5C 5D = 91/92/93) present in buffer but fragmented across extracted frames; when QLC+ Ch1 moved, Pico reported Ch14 firing
- **Analysis**: Searched for `0x00 0x00` pattern at offset +2 (standard DMX), but Enttec UART adds 13 extra bytes before channel data
- **Solution**: Changed extraction offset from +2 to +15 bytes after marker; validated with QLC+ test (Ch1→Ch14 offset proved +13 byte gap)
- **Lesson learned**: Enttec USB Pro's UART serial output includes firmware-specific metadata; XLR output follows standard DMX512 spec

### Problem 4: File Sync Issue with MicroPico
- **Symptom**: Modified code in VS Code but Pico kept running old version even after unplug/replug cycles
- **Cause**: Multiple .py files on Pico (main.py, main2024.py, run_pico.py, main2026_production.py) conflicting with upload
- **Solution**: Deleted all except main2026.py on Pico filesystem, forced fresh re-upload
- **Lesson learned**: Clean up old files before reloading; MicroPico may cache or load wrong file if multiples exist

## FINAL SOLUTION: Pattern-Based Frame Detection with First-Nonzero Anchor

**Problem Evolution:**
1. Timing-based detection failed (unreliable at fast polling speeds)
2. Searching for 0x00 markers found the WRONG zeros (break bytes instead of start code)
3. Offset calculations inconsistent because we couldn't reliably identify the real start code

**The Key Insight:**
The DMX break signal is encoded as ~13 zero bytes. Any channel data will have at least 3 non-zero bytes. Rather than searching for markers in a sea of zeros, **anchor on the guaranteed real data first**, then backtrack.

**Algorithm (Working):**
```
1. Find first NON-ZERO byte in buffer (guaranteed to be real channel data)
2. Verify position >= 1 (need room to backtrack)
3. The 0x00 at position (first_nonzero - 1) is the real start code
4. Extract 16 bytes from position (first_nonzero) onwards
5. Discard frame and repeat
```

**Why This Works:**
- Does NOT depend on timing
- Does NOT depend on pattern matching (even if there are anomalies)
- Anchors on guaranteed real data (channel values > 0)
- Robust across different DMX controllers and break byte counts
- Automatically skips all break bytes regardless of how many there are
